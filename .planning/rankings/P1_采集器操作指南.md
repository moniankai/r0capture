## 一句话说明

对红果 App `com.phoenix.read` 的排行榜页，挂 Frida hook 到 Compose 榜单 ViewModel 的
列表更新点 `SeriesRankTabViewModel.c0(List)`，从 item wrapper 层层反射到
`j30` 视频数据类，提取 `series_id / name / total / cover_url` 等字段并持久化。

Python 驱动 adb 完成导航 / swipe / 切榜编排，自动完成多榜采集。

## 一、核心数据通路

### Hook 点（与旧版 SaasVideoData 架构的根本区别）

新版红果排行榜页 `ShortSeriesKmpRankingActivity` 是 **Jetpack Compose** 实现，
**没有 RecyclerView / ViewHolder / SaasVideoData setter**。旧的 `resolve_interactive.py`
风格 hook (三 setter + holder.j2) 对这个页面完全无效（试采 0 条即此症）。

正确的数据通路：

```
SeriesRankTabViewModel.c0(List<tc4.e>)    ← 分页数据到达时调用 (hook 点)
                              │
   for item in List:          ▼
     tc4.e._a (y34.c 实例)   ← Frida 字段访问用 `_a.value`
                     │
                     ▼
     y34.c.g() → j30        ← 完整视频数据对象 (120+ 字段, 40 混淆 getter)
                     │
                     ▼
     拿出需要的字段 (见下表)
```

### j30 字段映射表（2026-04-20 ground truth 反推 + 修正版）

⚠️ **初版字段映射有 series_id/first_vid 互换的 bug** (见 METADATA_BUG_REPORT.md)。
下表为用已下载剧《疯美人》真 sid=`7624372698860227646` 作 ground truth 反推的**正确映射**:

| getter | 类型 | 含义 | 对应 dramas.json 字段 |
|--------|------|------|----------------------|
| `x()` | String (Long 化) | **series_id** 主键 (`u()` 等价, 作 fallback) | `series_id` |
| `E()` | String | **剧名** | `name` |
| `i()` | Integer | **总集数** | `total` |
| `g()` | String | **封面 URL** (heic) | `cover_url` |
| `A()` | String | **分类标签** (例: "古风爱情·全72集·张晋宜") | `recommend_text` |
| `J()` | String (Long 化) | **首集 biz_vid** (曾误认作 series_id 致 bug) | `first_vid` |
| `v()` | String (JSON) | 元数据 (含 batch 内 rank / channel_id / request_id) | 用于 `server_batch_pos` |
| `L()` | String | 置顶评论 | `top_comment` |
| `n()` | Integer | 热度/播放数值 | `popularity` |
| `e()` | String (#RRGGBB) | 主题色 | — |

> **关键**: `v()` 里的 `"rank": "N"` 是**当前 batch 内**的位置（1-10），
> 不是全局榜单排名。全局 rank_pos 必须由 Python 按首见顺序累计。

## 三、运行命令

### 默认（最常用）

```bash
python scripts/rank_collect.py --serial 4d53df1f --ranks "热播榜,漫剧榜"
```

- `--serial`: ADB 设备序列号（也可用环境变量 `ADB_SERIAL`）
- `--ranks`: 榜单列表逗号分隔。首版支持首屏可见 5 榜：
  **预约榜 / 推荐榜 / 热播榜 / 漫剧榜 / 新剧榜**
- 默认 `--per-rank-limit 0`（不限）和 `--max-swipes 40`

### 小剂量试采（新环境验证）

```bash
python scripts/rank_collect.py --serial 4d53df1f --ranks "热播榜" \
    --per-rank-limit 20 --max-swipes 5
```

### 干净一次（清旧产物重跑）

```bash
rm -f .planning/rankings/dramas.json .planning/rankings/snapshots.jsonl
python scripts/rank_collect.py --serial 4d53df1f --ranks "热播榜,漫剧榜" \
    --max-swipes 15
```

### 输出重定向 + unbuffered（后台跑必用）

```bash
python -u scripts/rank_collect.py ... > d:/tmp/collect.log 2>&1 &
```

**关键: `-u` 强制 Python 不缓冲 stdout**。Windows Git Bash 管道会把 buffered 输出
全吃掉，断进程时 log 文件是空的。

## 四、产物结构

### `.planning/rankings/dramas.json`

主档，以 `series_id` 为 key 的 dict。覆盖写，字段 union merge。

示例条目：
```json
{
  "7605549552493399064": {
    "series_id": "7605549552493399064",
    "first_seen_at": "2026-04-20T17:16:59",
    "source_ranks": ["漫剧榜/11", "热播榜/1"],
    "name": "从嬴政开始：历代皇帝陆续降临",
    "total": 40,
    "first_vid": "7605538863909440537",
    "cover_url": "https://p3-reading-sign.fqnovelpic.com/novel-pic/...",
    "recommend_text": "穿越·全40集",
    "top_comment": "史上第一位始皇帝嬴政逝世后, 被传送到一个奇幻之境...",
    "popularity": 2773125,
    "is_locked": null,
    "unlocked_eps": null,
    "last_updated_at": "2026-04-20T17:16:59"
  }
}
```

**字段 merge 规则**:
- `name` / `total` / `first_vid` / `cover_url` / `recommend_text` / `top_comment`: 空值不覆盖，首次见到就记下
- `source_ranks`: **UNION set**, 多榜累加不覆盖
- `first_seen_at`: 首次写入时刻, 以后不改
- `last_updated_at`: 每次合并都更新
- `is_locked` / `unlocked_eps`: 本轮信号不足, 留 `null` (P3 补)

### `.planning/rankings/snapshots.jsonl`

Append-only 快照, 一行一条。同一剧同榜多次采集会多行记录（追踪霸榜变化）。

```json
{"ts":"2026-04-20T17:16:59","rank_type":"热播榜","rank_pos":1,"server_batch_pos":1,"series_id":"7605549552493399064","session_id":"sess_20260420_171656"}
```

- `rank_pos`: **Python 累计首见顺序**（权威全局位置）
- `server_batch_pos`: 服务端 `meta_json.rank` 原值（**每批 1-10**, 仅供辅助参考）
- `session_id`: 当次运行 ID, 格式 `sess_YYYYMMDD_HHMMSS`

### 直接喂给 `hongguo_batch_lean.py`

```python
import json
dramas = json.load(open('.planning/rankings/dramas.json', encoding='utf-8'))
batch_input = [
    {"name": d["name"], "series_id": d["series_id"], "total": d["total"]}
    for d in dramas.values()
    if not d.get("is_locked") and d["total"] <= 150
]
```

## 五、脚本内部工作流程

```
main()
  ├─ 解析 CLI
  ├─ 自动发现 ADB 设备
  ├─ RankCollector.attach()
  │    ├─ 找 App pid
  │    ├─ frida.attach(pid)
  │    ├─ session.create_script(JS_HOOK)
  │    │    └─ JS: hook c0 + 解析 j30 + send batch 事件
  │    ├─ script.load() + 等 ready 事件
  │    └─ 开 on_message 线程: events queue 填充
  │
  └─ RankCollector.run(ranks)
       └─ for rank in ranks:
             ├─ nav_to_rank_activity(force_reenter=True)
             │    └─ BACK 到主页 → tap 剧场 (324,1820) → tap 排行榜 (442,381)
             │         (重试 3 次, 确保 focus == ShortSeriesKmpRankingActivity)
             │
             └─ collect_one_rank(rank)
                  ├─ sleep 1.5s  # 让 reenter 的 stale c0 事件全部入队
                  ├─ drain queue  # 清掉推荐榜/上次 tab 数据
                  ├─ tap_rank_tab(rank)  # 坐标见下表
                  ├─ sleep 2.5s  # 等 Compose 首屏加载完成
                  ├─ drain()  # 采首屏 (通常 10 条)
                  └─ while 未到限 && 未 3 轮无新:
                       ├─ adb_swipe (540,1550)→(540,900) 700ms
                       ├─ sleep 1s
                       └─ drain()  # 分页追加
                  └─ save_dramas(DRAMAS_JSON)  # 每榜结束保存
```

### 坐标 (1080x1920 竖屏)

| 用途 | 坐标 |
|------|------|
| 底部 剧场 tab | (324, 1820) |
| 剧场页 排行榜 入口 | (442, 381) |
| 预约榜 tab | (120, 516) |
| 推荐榜 tab (默认进入) | (324, 516) |
| 热播榜 tab | (528, 516) |
| 漫剧榜 tab | (732, 516) |
| 新剧榜 tab | (936, 516) |
| 下滚 swipe | (540,1550) → (540,900), 700ms |

## 六、故障恢复流程

### 症状 1 — `frida.TransportError: timeout was reached`（create_script / load 超时）

原因: frida-server 辅助 socket 进程残留, 或 Java bridge 被占。

处理:
```bash
# 1. 杀掉所有 Python 进程 (Frida session 可能残留)
taskkill //F //IM python.exe

# 2. 重启 frida-server (需要 root)
adb -s 4d53df1f shell "su -c 'killall -9 frida-server 2>/dev/null; sleep 1; nohup /data/local/tmp/frida-server > /dev/null 2>&1 &'"

# 3. 确认新 PID
adb -s 4d53df1f shell "su -c 'pidof frida-server'"

# 4. 验证通信
cd d:/dev/cursor/r0capture && python -u scripts/frida_ping.py
```

### 症状 2 — `Application Not Responding` 对话框（ANR）

原因: 挂了高频 hook (`okhttp3.Chain.proceed` 等)阻塞 main thread。

处理:
```bash
adb -s 4d53df1f shell am force-stop com.phoenix.read
sleep 2
adb -s 4d53df1f shell "monkey -p com.phoenix.read -c android.intent.category.LAUNCHER 1"
# 然后重启 frida-server (见症状 1)
```

**教训**: **不要挂 `okhttp3.internal.http.RealInterceptorChain.proceed`**，即使在 JS 内做
 URL filter 也会堵塞。只挂 ViewModel 的数据更新点 `c0(List)`。

### 症状 3 — `未进入排行榜页, 当前 focus: MainFragmentActivity`

原因: tap 剧场和排行榜入口间隔太短, tap 在过渡动画中 miss。

处理: 脚本已有 3 次重试（每次 sleep 2.2s + 3s）。还失败则：
1. 手动截图看当前 App 状态: `adb shell screencap -p /sdcard/x.png`
2. 检查是否有弹窗 / 广告 / 权限对话框拦截 tap

### 症状 4 — 某榜采集 0 条

可能原因与排查:

| 现象 | 原因 | 处理 |
|------|------|------|
| 上次采集结束时列表滚到底 | tab bar 被滚出 y=516, tap 打到剧卡进入播放页 | 脚本已加 `force_reenter` 每榜, 确保列表回顶 |
| hook 未触发 | 榜单 tab 可能是 static 不用 ViewModel, 或 App 缓存 | screencap 看页面状态 + 看 log 有无 `[JS err]` |
| App 崩溃 / crash | pid 已变 | 重新运行脚本, 会重新 attach 新 pid |

### 症状 5 — `RuntimeError: hook ready 超时`

JS 脚本 load 超时 10s 没收到 `ready`。大概率是 `Java.perform` 里某个 `Java.use(...)`
抛异常卡住。检查 App 是否处于**启动中**（闪屏/广告），等 App 完全进入主界面再运行。

## 七、已知局限

1. **漫剧榜实际只有 ~10 条**，之后 App fallback 显示"热播榜推荐"。现状这些 fallback
   条目会被标记为 "漫剧榜/11.." 进入 source_ranks。消费端如需过滤真实漫剧榜，用
   `source_ranks` 里 `漫剧榜/pos ≤ 10` 判定。
2. **热播榜 15 次 swipe 停止时常 `max_swipes`**，实际可能还有更多分页。加大
   `--max-swipes 30` 或 `40` 再试。
3. **横滑才能看到的榜**（演员榜 / 必看榜 / 收藏榜 / 热搜榜）首版**未支持**。
   需要扩 `RANK_TAB_X` + 加横滑逻辑（参考 [P0_探测结果.md](P0_探测结果.md) 的横滑坐标）。
   演员榜还需跳过或特殊处理（按演员聚合, SaasVideoData getter 拿不到 series_id）。
4. **`is_locked` / `unlocked_eps` 字段留 null**。j30 有 40+ boolean/int getter
   含义未细标, P3 阶段再 dump + 肉眼对比反推（有 `H()` `I()` `d()` `m()` `y()` `z()`
   等返回 Boolean 的 getter 可能与 VIP / 广告锁相关）。
5. **rank_pos 不是绝对权威**: 使用 Python 首见顺序, 若 RecyclerView prefetch
   导致绑定顺序 ±1, rank_pos 会有微小偏差。HANDOFF schema 的 `source_ranks`
   是 UNION list 不强依赖精确值, 可接受。

## 八、扩展方向（给后续开发）

### 支持横滑后的榜（预约榜之外的其他）

加坐标到 `RANK_TAB_X`, 改 `tap_rank_tab()` 接收 `swipe_count_before_tap` 参数:
```python
RANK_TAB_LAYOUT = {
    "预约榜":  (120, 516, 0),
    "演员榜":  (390, 516, 1),  # 先横滑 1 次再 tap
    "必看榜":  (594, 516, 1),
    "收藏榜":  (798, 516, 1),
    "热搜榜":  (1002, 516, 1),
    ...
}
```

横滑: `swipe(900, 516, 200, 516, 600)` 每次露出 3 个新 tab。

### 增加 is_locked 探测

在 j30 getter dump 脚本（`scripts/rank_probe_j30.py`）里加一列"已知剧是否付费"
对比 ground truth 反推哪个 getter 返回 VIP 锁状态。

### 增加二级分类分流（女频 / 男频 / 分类筛选）

tap `close_down` 按钮展开筛选面板, tap 分类标签（见 P0 的标签坐标表）
→ 榜单重载 → c0 再次触发 → 对应分类数据。
